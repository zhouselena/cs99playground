# Pokémon GO Infrastructure rscore Report

## 1. Post (Current) Infrastructure Summary

### What Was Modeled

The post infrastructure represents Pokémon GO's current production architecture (~2024-2025) as operated by Niantic. Pokémon GO is one of the most spatially demanding mobile games ever built: a real-time augmented-reality game with 90M+ monthly active players that requires continuous GPS processing, server-side spawn computation per player location, raid matchmaking, real-time Gym and PvP battle resolution, and global live events (Community Days, Pokémon GO Fest) with massive concurrent player spikes. The graph contains **70 nodes** (50 functional, 20 provider) and ~161 edges.

### Key Architectural Decisions

**GKE Autopilot as Primary Compute**
Niantic runs on Google Cloud Platform — a natural fit given they spun out of Google's internal incubator. The vast majority of backend services run on `GKE Autopilot` (0.78), Niantic's managed Kubernetes layer. GKE Autopilot removes node management overhead and auto-scales workloads, which is critical for Pokémon GO's highly variable traffic (Community Days can spike player activity 5–10× overnight). Niantic has confirmed GKE usage at Google Cloud Next events.

**Cloud Spanner for Globally Consistent Player Data**
Player profiles, inventory, friends lists, and trading state are stored in `GCP Cloud Spanner` (0.92). Niantic confirmed Cloud Spanner usage at Google Cloud Next, citing its ability to serve globally consistent reads and writes without sacrificing horizontal scalability — critical when a player in Tokyo trades Pokémon with a player in São Paulo and both inventories must be atomically updated. Cloud Spanner's 99.999% SLA and external consistency make it the right choice for a game where item duplication bugs destroy the economy.

**Firestore for Game State and Pokemon Spawns**
Real-time game state (active sessions, per-map Pokémon spawn state, research task assignments) and configuration data live in `GCP Firestore` (0.82). Firestore's document model and real-time listeners suit the spawn system well: Pokemon Spawn Service writes spawn events to Firestore, which Location Service reads to determine what appears on each player's map.

**Cloud Pub/Sub for Event-Driven Messaging**
`GCP Cloud Pub/Sub` (0.82) underpins the Event Bus, Telemetry Collector, and Analytics Pipeline. Pub/Sub's global replication and at-least-once delivery semantics handle the high-volume event streams generated by location updates, battle completions, raid joins, and PokéStop spins.

**Cloud Run for Serverless Stateless Workloads**
Lightweight stateless functions — `Weather Service` (syncing in-game weather to real-world conditions via weather APIs) and `Adventure Sync Service` (processing pedometer data from iOS Health/Google Fit) — run on `GCP Cloud Run` (0.78). These workloads are low-frequency, event-triggered, and don't warrant dedicated Kubernetes pods.

**Cloudflare Enterprise + Shield Advanced for DDoS Defense**
Niantic has been a repeated DDoS target (most famously during the 2016 launch chaos). `Cloudflare Enterprise` (0.88) provides edge CDN and layer-7 DDoS absorption, while `AWS Shield Advanced` (0.87) protects the API Gateway layer. `AWS WAF (Managed Rules)` (0.75) adds application-layer protection. This three-layer DDoS stack reflects the defense-in-depth appropriate for a platform with high-profile public events.

**AWS Global Accelerator for Low-Latency Global Routing**
`AWS Global Accelerator` (0.87) provides Anycast routing for the Load Balancer and API Gateway, ensuring mobile clients worldwide reach the nearest AWS edge before being routed to Niantic's GCP backend. This reduces latency for the location update → spawn computation → response cycle, which is latency-sensitive (players see stale maps otherwise).

**Raid and Remote Raid Social Layer**
`Raid Service` depends on `Battle Service`, `Matchmaking Service`, and `Game State Service`. The `Remote Raid Service` — launched in April 2020 and a permanent feature since — additionally depends on `Friends Service` and `Party Service`, modeling the social graph required to invite and join raids remotely. This two-service design (local Raid vs. Remote Raid) reflects Niantic's architecture where remote raids have additional social prerequisites.

**ElastiCache for Session and Matchmaking State**
`AWS ElastiCache (cluster mode)` (0.70) backs `Session Manager` (active player sessions) and `Matchmaking Service` (raid and PvP matchmaking pools). Cluster mode's sharding and automatic failover prevents session loss during traffic spikes.

**Stripe for In-App Purchases**
`Payment Gateway` (fiat → PokéCoins conversion) and `Shop Service` are both backed by `Stripe Payments` (0.88). Niantic uses Stripe for purchase processing across iOS and Android; the Shop Service and Payment Gateway both connect to Stripe to ensure purchases are idempotent and reconcilable.

**Observability: Datadog Enterprise + CloudWatch + OpenSearch**
Three-layer observability: `AWS CloudWatch (Container Insights)` (0.75) for GKE/ECS-native metrics; `Datadog Enterprise` (0.82) for full-stack APM wired to both `Monitoring Service` and `Telemetry Collector`; `OpenSearch (multi-AZ with standby)` (0.75) for log analytics powering Analytics Pipeline and Monitoring Service. `PagerDuty` (0.80) handles on-call routing.

### Why This Is a Best-Guess of Reality

Niantic has publicly confirmed: GCP as their primary cloud provider (Google Cloud Next keynote appearances), Cloud Spanner for player data, GKE for container orchestration, and Cloudflare for CDN/DDoS. The raid system (local and remote), weather integration, Adventure Sync, research tasks, and the Pokémon GO shop are all documented game features. The DDoS protection stack reflects the real incidents Niantic experienced at launch and during major events. AWS services (Global Accelerator, ElastiCache, Shield Advanced, CloudFront) are used as complementary edge services layered on top of the GCP core.

## 2. Pre (Degraded) Infrastructure — Degradations Explained

The pre infrastructure represents Pokémon GO at launch (~2016), during Niantic's infamous day-one scaling disaster where servers buckled under 50× more traffic than anticipated. It keeps the same 50 functional nodes but uses only **18 provider nodes** (removed AWS Global Accelerator and Cloudflare Enterprise), with provider quality downgrades, and removes 12 functional dependency edges.

### Provider Downgrades

| Component | Pre | Post | Score Impact |
|---|---|---|---|
| Compute | `GKE Standard (multi-zone)` 0.75 | `GKE Autopilot` 0.78 | -0.03 |
| Player DB | `Cloud SQL (HA)` 0.68 | `Cloud Spanner` 0.92 | -0.24 |
| Cache | `ElastiCache (single node)` 0.35 | `ElastiCache (cluster mode)` 0.70 | -0.35 |
| DDoS | `Shield Standard` 0.60 | `Shield Advanced` 0.87 | -0.27 |
| CDN/Edge Layer | CloudFront only | CloudFront + Cloudflare Enterprise | Reduced redundancy |
| Global Routing | ALB only | ALB + Global Accelerator | Reduced Anycast reach |
| Object Storage | `S3 One Zone-IA` 0.55 | `S3 Standard` 0.85 | -0.30 |
| Observability | `CloudWatch (standard)` 0.72 | `CloudWatch (Container Insights)` 0.75 | -0.03 |
| APM | `Datadog Pro` 0.75 | `Datadog Enterprise` 0.82 | -0.07 |
| Log Analytics | `OpenSearch (single AZ)` 0.55 | `OpenSearch (multi-AZ with standby)` 0.75 | -0.20 |

The Cloud SQL (HA) downgrade for the player DB is the most architecturally significant: a single-region SQL database cannot provide the global active-active consistency that Cloud Spanner offers, creating regional consistency windows for player profile and inventory updates.

### Topology Removals (12 edges removed)

1. **Remote Raid Service → Friends Service** — Remote Raids launched April 2020; not in the original infrastructure
2. **Remote Raid Service → Party Service** — same; party-based remote raid invites didn't exist at launch
3. **Analytics Pipeline → Event Bus** — analytics was batch-based at launch, not event-driven
4. **Telemetry Collector → Location Service** — location telemetry was not wired to the telemetry pipeline at launch
5. **Gift Service → Friends Service** — Gifts launched July 2018; not at launch
6. **Adventure Sync Service → Player Profile Service** — Adventure Sync launched October 2018; profile integration missing at launch
7. **Leaderboard Service → Player Profile Service** — leaderboards initially showed simple stats without full profile integration
8. **Moderation Service → Anti-Cheat Service** — moderation was manual/disconnected from anti-cheat at launch
9. **Research Service → Event Bus** — Field Research launched March 2018; event-driven research completion wasn't present at launch
10. **Event Service → Event Bus** — early event system was not event-driven; updates were batch-polled
11. **Trading Service → Anti-Cheat Service** — Trading launched June 2018; anti-cheat wasn't integrated with trading initially
12. **News Feed Service → Event Service** — news/events weren't integrated with the event routing system at launch

These removals target:
- **Post-launch feature absence**: Remote Raids, Gifts, Adventure Sync, Trading, Research, and Field Events are all features that launched 6 months to 4 years after the game.
- **Triangle-breaking**: Removing Gift → Friends breaks the Gift → Inventory → (Friends → Account) triangle. Removing Leaderboard → Player Profile breaks the Leaderboard → Player Stats → Player Profile triangle.
- **Event-driven decoupling**: Removing Analytics → Event Bus and Event Service → Event Bus disconnects the event-driven subsystem.
- **Cross-cluster edge removal**: Removing Telemetry → Location, Moderation → Anti-Cheat, and Research → Event Bus weakens the cross-cluster algebraic connectivity.

### Root Cause of the Anomalies

**Degree Entropy decrease**: POST adds two provider nodes (Cloudflare Edge and AWS Accelerator) each with exactly 2 consumer nodes. These additions plus the increased in-degree of multi-hosted nodes (API Gateway rises from 3 to 4 hosting providers) concentrate the degree distribution at specific values rather than broadening it. Shannon entropy decreases when a distribution becomes more peaked at particular values, which is what happens here.

**Betweenness decrease**: This mirrors the pattern seen in Arc Raiders. POST's 12 new dependency edges — especially the social cluster additions (Remote Raid → Friends, Gift → Friends, Party → Friends creating a Friends-hub triangle) and the event cluster additions (Research → Event Bus, Event Service → Event Bus, PokeStop → Event Bus, Analytics → Event Bus making Event Bus a high-centrality relay) — route substantially more shortest paths through Friends Service (criticality 0.0870 PRE → 0.1131 POST) and Anti-Cheat Service (criticality 0.0810 PRE → 0.1152 POST). The concentration of new shortest paths through these emerging hubs raises their raw betweenness centrality, which reduces the displayed (1-betweenness) score. The four other metrics' improvements more than compensate: POST wins overall by +0.0108.

### Node Criticality Observations

**GCP Compute** criticality falls sharply from 0.5314 (PRE) to 0.2269 (POST). In PRE, `Cloud SQL (HA)` and the lack of edge providers (no Cloudflare, no Global Accelerator) force a higher fraction of shortest paths through GCP Compute. In POST, Cloud Spanner takes over data hosting for five key services, Cloudflare Edge and Global Accelerator absorb routing paths at the edge, and the 12 additional functional edges create alternative routing through the social and event subgraphs — all reducing GCP Compute's relative centrality dramatically.

**API Gateway** criticality collapses from 0.3220 (PRE) to 0.0205 (POST). In PRE, API Gateway is a critical chokepoint because most inter-service paths must route through it. In POST, two factors reduce its centrality: (1) it gains a fourth hosting provider (AWS Accelerator) increasing its structural redundancy, and (2) the 12 new functional edges create many more direct inter-service paths (Remote Raid → Friends → Account; Gift → Friends → Account; Research → Event Bus → Analytics) that bypass the API Gateway entirely.

**Event Bus** criticality rises from 0.0348 (PRE) to 0.0925 (POST). In POST, four additional edges connect to Event Bus (Research → Event Bus, Event Service → Event Bus, PokeStop → Event Bus already in both, and Analytics → Event Bus). This makes Event Bus a genuine relay node in the event-driven subgraph, a structurally accurate reflection of its importance in Pokemon GO's live-event and progression architecture.

**AWS Cache (ElastiCache)** criticality drops from 0.2308 (PRE) to 0.1783 (POST) — the single-node ElastiCache in PRE is a more structurally critical SPOF than the cluster-mode version in POST, which has more resilient internal replication and therefore appears as a less concentrated bottleneck in betweenness terms.
